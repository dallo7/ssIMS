1What only you can do

Every other role can do the day job. You do the things that change the rules.

UsersCreate, disable, re-role and reset PINs for any user.
ConfigCategories, units of measure, currencies, integration keys.
AuditingThe full event log of every transaction in the system.
MonitoringSystem health, queue lag, error stream.
Reverse shipmentsOnly ADMIN can undo an SO shipment.
Database backupsSnapshot, export and restore at the OS level.
ℹ︎
You are the safety net. Clerks record movement, managers approve and oversee, and you make sure the system itself stays trustworthy. When something cannot be fixed through normal flow, the manager calls you.

3Managing users

Create, re-role, deactivate and reset credentials. All actions are logged.

Create a user

  1. Open Users from the sidebar.
  2. Press + New user.
  3. Fill in:
    • Username — short, no spaces, lowercase. Used for login.
    • Full name — appears in audit logs and on the dashboard greeting.
    • Role — pick ADMIN only with extreme care; default to STOCK_CLERK.
    • Initial password — tell the user privately and force change at first login.
    • PIN — 4–6 digits, used to sign approvals (managers and admins only).
  4. Press Save.

Change a role or reset access

  1. Find the user in the table at the bottom of the Users page (filter by name).
  2. Click their row. The right-hand panel exposes Role, Reset password, Reset PIN, Active.
  3. Make the change and press Save. The user is forced to re-authenticate at next request.
⚠
Never delete users with history. Toggle Active off instead. Deletion would orphan their audit entries.
✕
At least two ADMIN accounts at all times. If the only admin is locked out, you'll need OS-level access to recover. Always keep a backup admin (a personal one and a shared "ops-admin" emergency one stored in the safe).

4System configuration

The values that shape every page in the app.

Open Config from the sidebar. You'll find:

  • Workspace name — shows in the header and on every printed document.
  • Default warehouse — pre-fills the warehouse dropdown when a clerk creates a bin.
  • Default currency — used for inventory value rollups.
  • Reorder defaults — fallback values when a new SKU is created without explicit reorder fields.
  • Integration API keys — endpoints that pull live data from external systems (e.g. accounting). Rotate these every 90 days.
⚠
Changing units of measure or categories? These are shared across every item that uses them. Always rename rather than delete; the database preserves IDs but the user-visible label updates everywhere.

5Master data hygiene

Categories, units of measure, suppliers, customers — keep them small, clear and consistent.

  1. Categories — keep the list under 25. If clerks are submitting one-off categories, talk to the manager about consolidating.
  2. Units of measure — define them in their canonical form (PIECE, BOX-12, KG, LITER). Do not create both kg and KG.
  3. Suppliers / Customers — managers maintain these, but you should audit quarterly for duplicates and missing contact details.
✓
Have one source of truth. When importing a list from Excel, do it once through Config → Bulk import, then disable the Excel sheet. Two parallel lists will diverge within a week.

6Auditing every action

If it happened in the system, it's here, and you can prove who did what and when.

Open Auditing from the sidebar. Each row records:

ColumnWhat it tells you
TimeServer timestamp (UTC) of the event.
ActorThe username (and role at the time).
Actione.g. STOCK_IN, PO_RECEIVE, SO_SHIP, USER_CREATE, APPROVAL.
SubjectThe item / order / user the action affected.
ReferenceWhatever the actor typed in the reference field.
IP / deviceWhere the request came from.

Common investigations

  • "Why is SKU X negative?" — Filter by item, sort by time, find the issue that wasn't preceded by a receipt.
  • "Who shipped that order yesterday?" — Filter by action SO_SHIP and the SO number.
  • "Who reset the manager's password?" — Filter by action USER_PIN_RESET or USER_PWD_RESET.
ℹ︎
Audit log is append-only. Nobody — not even ADMIN — can edit or delete an audit row through the UI. If you ever need to redact, do it via a documented database operation with a paper trail.

7Monitoring & alerts

Spot problems before users report them.

Open Monitoring from the sidebar. The page shows:

  • Health checks — DB connectivity, disk usage, response time, last successful backup.
  • Approval queue depth — if it's persistently >20, your manager is overloaded; consider promoting a second one.
  • Error stream — uncaught exceptions logged in the last 24h, grouped by URL.
  • Active sessions — who's logged in right now.

Acknowledging an alert

  1. Click the alert row to expand the details.
  2. Read the stack trace / context.
  3. If you've actioned it, press Acknowledge. Acknowledged alerts move to the bottom of the list and stop notifying.
  4. If it's still open after 24 hours and unacknowledged, the system pings on-call.

8Posting corrections that managers can't

Reversing shipments, adjusting historic cost layers, deleting bad master data.

Reversing a Sales Order shipment

  1. Open Sales Orders, find the SO and click its row.
  2. Press Reverse last shipment (only visible to ADMIN).
  3. Confirm with your PIN. The shipment is undone, stock is returned, and the audit log records the reversal.

Voiding a duplicate item

  1. Open Inventory, find the duplicate.
  2. Press Soft-delete. The item disappears from pickers but its history stays.
  3. If the duplicate had stock, transfer it to the canonical item via Movements first (Stock Out from duplicate, Stock In to canonical) so the on-hand stays correct.
✕
Never hard-delete an item or transaction. The buttons that suggest "delete" perform a soft delete that preserves history. Hard deletion can only be done with database access and should be logged on the operations runbook.

9Backups & data export

A backup you have not tested is not a backup.

Automated backups

The server runs a nightly snapshot of the SQLite database to /var/backups/ssims/ at 02:00 server time. Retention: 30 daily, 12 monthly, 7 yearly. The Monitoring page shows the timestamp of the last successful backup — if it's older than 26 hours, escalate.

Manual export from the UI

  • Reports → any grid → header menu → Export CSV.
  • Auditing → Export to CSV for legal / compliance dumps.

Restoring from backup

Restoration is an OS-level operation:

# on the server, as the deploy user
sudo systemctl stop ssims
sudo cp /var/backups/ssims/cpi_inventory-2026-04-17.sqlite \
        /opt/ssims/data/cpi_inventory.sqlite
sudo systemctl start ssims
⚠
Test the restore once a quarter. On the staging box. If the file is corrupt, you want to find out before you need it.

10Deploying updates

When the development team pushes a new release.

  1. Tell the team via WhatsApp / Slack: "deploying in 5 minutes, expect a 30-second blip".
  2. SSH into the server.
  3. Pull the new code and restart: 
    cd /opt/ssims
sudo -u ssims git pull origin main
sudo systemctl restart ssims
  4. Watch the service come back: 
    sudo systemctl status ssims
sudo journalctl -u ssims -n 50 --no-pager
  5. Open the app in your browser, hard-refresh (Ctrl+Shift+R), and click through the dashboard, Movements, Sales Orders, Approvals to verify they load.
  6. Announce: "deployed, all green".
✕
Never deploy on a Friday afternoon. If something breaks, you'll be on the phone all weekend. Deploy Tue–Thu mornings.

11Incident playbook

A repeatable response when something is on fire.

  1. Acknowledge. Reply to the alert / call within 5 minutes — even just "looking into it".
  2. Assess. Open Monitoring. Is the app up? Is the DB up? Is disk full?
  3. Contain. If it's a single user breaking things, deactivate them temporarily. If it's the whole app, restart: 
    sudo systemctl restart ssims
  4. Communicate. Tell users in plain language what's happening and your ETA.
  5. Resolve. Apply the fix (rollback, hotfix, configuration change).
  6. Post-mortem. Within 48 hours write a one-page note: what happened, why, what you changed, what we'll do so it doesn't recur. File it.

Quick rollback to the previous release

cd /opt/ssims
sudo -u ssims git log --oneline -5      # pick the last good SHA
sudo -u ssims git checkout <sha>
sudo systemctl restart ssims
ℹ︎
If the database is corrupted stop the service, restore from last night's backup, and run a delta from the audit log to replay anything that happened since the snapshot.

12Best practices for admins

Five rules that separate quiet weeks from chaotic ones.

  1. Principle of least privilege. Default new users to STOCK_CLERK or VIEWER. ADMIN is for two people, max.
  2. Read the audit log every Monday. 10 minutes. Spot a clerk over-issuing or a manager rubber-stamping before it becomes a quarter-end issue.
  3. Backups, restores, drills. Backups run themselves; restores need to be rehearsed.
  4. Document deploys and incidents. A markdown file in docs/runbook/ beats memory every time.
  5. Stay out of the SQL prompt during business hours. Anything you do there bypasses the audit log. Save it for after-hours and write it down.
